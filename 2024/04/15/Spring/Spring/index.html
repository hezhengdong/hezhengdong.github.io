SpringBean-生命周期

对之前IOC的总结

<span style="color:blue">两个Map集合

  1. BeanDefinitionMap存储单纯的bean标签对象

    • 通过reader将xml标签读取到beanDefinationMap中
    • 然后registry注册

  2. singletonObjects存储的是实例化好的对象

<span style="color:blue">Bean工厂后处理器

BeanFactoryPostProcessor Bean工厂处理器

通过重写方法，对beanDefination进行修改，也可以人为进行beanDefination的注册，只要将自定义的beanDefination人为添加到Map集合中就行然后Spring自动托管

因为BeanFactoryPostProcessor内部注册beanDefination有点麻烦，所以学习了它的子接口BeanDefinitionRegistryPostProcessor，内部提供的方法很方便向Map中注册beanDefination

然后Bean对象创建完毕之后，填充到Map之前，有一个BeanPostProcessor，Bean后处理器

<span style="color:blue">Bean后处理器

分为before方法和after方法

能够对Bean对象功能增强，比如通过动态代理增加一些日志、进行包装，然后将Proxy增强对象放到singletonObjects当中，

<span style="color:blue">左边部分做的是对beanDefinitionMap的填充，bean的定义过程

<span style="color:blue">右边部分

  • bean的实例化阶段
  • bean的初始化阶段
  • bean的存储阶段

<span style="color:blue">生命周期

整合到一起叫做Spring Bean的生命周期

  • 实例化创建对象（半成品对象）
  • 属性填充（依赖注入）
  • 再看当前的bean是否需要实现Aware接口（一般用不到，高级部分常用到的）（Spring帮你注入一些你需要的底层的bean）
  • BeanPostProcessor before方法
  • InitializingBean接口初始化方法，afterPropertySet方法（见名知意，在属性填充之后执行）
  • 再执行自定义的初始化方法init
  • BeanPostProcessor after方法
  • 然后将最终的对象存储到singletonObjects容器当中

1.3 原理

  • SqlSessionFactoryBean负责产生SqlSessionFactory，并将SqlSessionFactory放到bd容器中，谁用就依赖注入谁
  • MapperScannerConfigurer实现了BeanDefinitionRegistryPostProcessor, InitializingBean, ApplicationContextAware, BeanNameAware这四个接口，其中 <span style="color:blue">BeanDefinitionRegistryPostProcessor，将UserMapper接口封装为对象，然后存进map结合中
  • MapperScannerConfigurer扫描的入口，加载配置文件，扫描包，扫描的时候，调用ClassPathMapperScanner，扫完后，将BeanDefinition放到beanDefinitionMap中，再通过ClassPathMapperScanner修改bean的全限定类名，修改成MapperFactoryBean的，最后在getBean是调用的是Mapper的FactoryBean的getObject方法，getObject方法调用的优势mybatis的SqlSession.getMapper，最终把一个Mapper方法放到了Spring容器中

  • xml配置组件扫描：

  • 配置类配置组件扫描：

只要将Bean对应的BeanDefinition注册到beanDefinitionMap中，就可以经历整个SpringBean的生命周期，最终实例化进入单例池中。

所以无论是xml还是注解方式，底层都是要将数据封装为beanDefinition对象。

6.1 基于xml方式的组件扫描

入口：<context:component-scan base-package=”com.ithzd”/>

在自定义命名空间处，扫描到@component注解，然后把解析到的beanDefinition存储到了beanDefinitionMap当中。（通过BeanFactoryProcessor来注册）

6.2 基于注解方式的组件扫描（配置类）

  • 通过AnnotatedBeanDefinitionReader来进行对注解的读取解析
  • AnnotatedBeanDefinitionReader里面调用了AnnotationConfigUtils.registerAnnotationConfigProcessors
  • 里面有一个set集合，里面再去调用registry.registryBeanDefinition(beanName,definition)
  • 然后将ConfigurationClassPostProcessor（实现了BeanDefinitionRegistryPostProcessor）注册到beanDefinitionMap当中。

添加过程中，又回到了与xml方式相同的ClassPathBeanDefinitionScanner.doScan方法中进行扫描。所以不管是xml方式还是注解方式，最终它们到达的扫描方法是一致的。

3.2.1 <span style="color:blue">平台事务管理器

PlatformTransactionManager是Spring提供的封装事务具体操作的规范接口，封装了事务的提交和回滚方法

不同的持久层框架事务操作的方式有可能不同，所以不同的持久层框架有可能会有不同的平台事务管理器实现， 例如，MyBatis作为持久层框架时，使用的平台事务管理器实现是DataSourceTransactionManager。 Hibernate作为持久层框架时，使用的平台事务管理器是HibernateTransactionManager。

3.2.2 <span style="color:blue">事务定义信息配置

每个事务有很多特性，例如：隔离级别、只读状态、超时时间等，这些信息在开发时 可以通过connection进行指定，而此处要通过配置文件进行配置。

切点表达式指定的方法与此处指定的方法的区别

  • 切点表达式，是过滤哪些方法可以进行事务增强；
  • 事务属性信息的name，是指定哪个方法要进行哪些事务属性的配置

1. name

name属性名称指定哪个方法要进行哪些事务的属性配置

方法名在配置时，也可以使用* 进行模糊匹配，例如：

2. isolation

isolation属性：指定事务的隔离级别，事务并发存在三大问题：脏读、不可重复读、幻读/虚读。可以通过设置事 务的隔离级别来保证并发问题的出现，常用的是READ_COMMITTED 和REPEATABLE_READ

3. read-only

设置当前的只读状态，如果是查询则设置为true，可以提高查询性能，如果是更新（增删改）操 作则设置为false

4. timeout

设置事务执行的超时时间，单位是秒，如果超过该时间限制但事务还没有完成，则自动回滚事务 ，不在继续执行。默认值是-1，即没有超时时间限制

5. propagation

设置事务的传播行为，主要解决是A方法调用B方法时，事务的传播方式问题的，例如：使用 单方的事务，还是A和B都使用自己的事务等。事务的传播行为有如下七种属性值可配置

web层MVC框架思想与设计思路

总结 整理！！！

复习Javaweb的一些知识：

A、复习Javaweb三大组件，三大组件的特点，三大域对象，共同点，都能在业务逻辑之前插入操作。

B、复习servlet：

  1. 客户端发送一个Http请求
  2. 然后web服务器tomcat接收请求
  3. 根据url转发给对应的servlet
  4. 然后就是servlet的生命周期（如果是首次请求某个Servlet，Web服务器会先加载这个Servlet类，创建其实例，并调用 init方法进行初始化。如果Servlet已经被加载和实例化，这一步将被跳过）
  5. 然后根据http请求类型创建调用对应的方法，创建HttpServletRequest对象和HttpServletResponse对象并传递给方法
  6. 执行业务逻辑，发送响应，servlet销毁

现在想要干的事：

  • 将Servlet与Spring容器集成，旨在让Servlet能够使用Spring容器管理的Bean，以便利用Spring提供的众多特性，如依赖注入、事务管理等。

做法：

如果直接在控制层中对每个servlet创建一个applicationContext，多个Spring容器，不满足单例，而且浪费内存。

所以applicationContext要统一创建，综合比较三大组件，选择监听器。选择监听器（监听器没学过，存疑），在Servlet域对象中创建一个统一的applicationContext容器，保证Spring创建的Bean是单例的，以后使用时直接在ServletContext中调用已创建好的applicationContext

又一个问题：

Servlet中的一部分功能属于共有功能，所以就引入了DispatcherServlet

只创建一个Servlet：这句话的意思可能比较模糊。在典型的Spring Web应用中，通常至少会有一个 DispatcherServlet实例，它是Spring MVC的核心。在一些特定场景下，可能通过配置将所有请求都路由到这一个 DispatcherServlet，由它负责分发到不同的控制器（Controller）。但这并不意味着Web应用中只有这一个Servlet，因为还可能有其他用途的Servlet存在，例如处理静态资源。

Servlet域对象和监听器在其中发挥的作用到底是什么?

ServletContextListener：监听 ServletContext 的生命周期事件，如 ServletContext 的创建和销毁。可以用来初始化资源和清理资源。
